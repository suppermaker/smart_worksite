# 智慧工地大模型应用系统-架构设计文档

版本：v1.0
状态：草稿
架构形态：模块化单体架构，可演进为微服务架构
技术栈建议：后端 Java + Spring Boot / 前端 Vue 3 + TypeScript / 智能算法服务 Python / MyBatis / MySQL / Redis / MinIO / Milvus或pgvector / 大模型服务 / OCR服务
适用范围：智慧工地知识问答、合规审查、报告生成、OCR识别系统开发

## 1. 文档概述

### 1.1 文档目的

本文档基于《智慧工地大模型应用系统-需求文档》进行架构设计，明确系统边界、模块划分、核心流程、数据存储、外部能力集成、安全机制、任务编排和异常处理机制。

本文档用于指导详细设计、接口设计、数据库设计、编码实现和系统部署。

### 1.2 系统定位

智慧工地大模型应用系统是面向建筑工地管理场景的智能应用平台。系统通过接入项目资料、政策标准、业务数据库和模型能力，提供知识问答、合规审查、报告生成、OCR识别等能力。

系统本身不限定某一个固定大模型，而是通过适配层统一接入大模型、RAG知识库、数据库问答、OCR引擎等外部或内部智能能力。

### 1.3 架构目标

1. 支持智慧工地多项目、多知识库、多数据源隔离管理。
2. 支持知识问答、合规审查、报告生成和OCR识别四大应用能力。
3. 支持Word、PDF、Excel、PPT、图片等多格式数据接入。
4. 支持模型、知识库、数据库之间的智能路由和混合检索。
5. 支持异步任务、状态追踪、失败重试和全过程留痕。
6. 支持统一权限、文件访问控制、敏感信息脱敏和审计日志。
7. 保证系统具备可扩展性、可观测性、稳定性和可维护性。

## 2. 架构约束与设计原则

### 2.1 架构约束

- 系统一期建议采用模块化单体架构，降低开发和部署复杂度。
- 数据必须按工地项目隔离，禁止跨项目访问知识库、报告和文件。
- 耗时任务必须异步执行，避免HTTP请求长时间阻塞。
- 外部模型、知识库、OCR、数据库查询等能力必须通过适配层调用。
- 所有接口统一响应结构、统一错误码和统一链路追踪。

### 2.2 设计原则

#### 2.2.1 模块化单体

在一个后端应用内按业务领域拆分模块，每个模块拥有独立Controller、Application Service、Domain、Repository、Mapper和DTO。模块之间通过服务接口交互，避免跨模块直接访问内部Mapper或数据表。

#### 2.2.2 领域边界清晰

系统划分为项目权限、文件数据、知识库、模型适配、问答、合规审查、报告生成、OCR、任务编排、审计安全等模块。每个模块只负责自己的领域能力。

#### 2.2.3 外部能力适配隔离

大模型、RAG知识库、OCR引擎、业务数据库、对象存储等能力统一通过Adapter封装。业务模块不直接感知外部服务URL、鉴权方式、请求格式和重试逻辑。

#### 2.2.4 异步任务驱动

知识库入库、报告生成、OCR批量识别等任务采用异步队列处理。前端通过任务状态接口查询进度，避免接口超时。

#### 2.2.5 全链路可追踪

系统对用户请求、文件解析、知识库检索、模型调用、数据库查询、OCR识别、报告渲染和人工修改过程进行留痕，便于审计和问题排查。

系统整体分为五层：

| 层级 | 说明 |

## 4. 应用分层设计

### 4.1 分层结构

```text
Controller 层
  ↓
Application Service 层
  ↓
Domain Service / Domain Model 层
  ↓
Repository / Mapper 层
  ↓
MySQL / Redis / MinIO / 外部服务
```

### 4.2 各层职责

| 层级 | 职责 |
| --- | --- |
| Controller | 接收HTTP请求、参数校验、调用应用服务、返回统一响应 |
| Application Service | 编排业务用例，处理事务边界，调用领域服务和适配器 |
| Domain Service | 封装核心业务规则，如权限判断、任务状态流转、审查规则 |
| Repository/Mapper | 负责数据库读写，不处理复杂业务规则 |
| Adapter | 封装外部模型、知识库、OCR、对象存储、业务数据库等依赖 |

## 5. 模块设计

### 5.1 项目权限模块

- 用户、角色、权限、项目成员管理。
- 登录认证、令牌校验、当前用户上下文。
- 项目级数据权限过滤。
- 文件访问权限和接口权限校验。

### 5.2 文件与数据模块

- 文件上传、下载、删除、预览地址生成。
- 文档解析，支持Word、PDF、Excel、PPT、图片。
- 数据清洗、去重、标准化和质量校验。
- 原始文件、解析结果、标准化数据版本管理。

### 5.3 知识库模块

- 项目级知识库创建、启停用和删除。
- 文档入库、切片、向量化和来源记录。
- 知识库检索、TopK召回、重排序和引用返回。
- 互联网政策资讯可配置采集和增量更新。

### 5.4 智能问答模块

- 问答会话管理。
- 智能路由：模型、知识库、数据库、混合检索。
- 多轮上下文和主动追问。
- 回答来源、引用资料和反馈记录。

### 5.5 合规审查模块

- 审查模板管理。
- 文档审查任务提交。
- 模型或规则服务审查。
- 问题定位、修改建议、JSON结果返回。
- 审查记录和审查状态管理。

### 5.6 报告生成模块

- 报告模板管理和模板变量解析。
- 报告生成配置管理。
- 多数据源填充：模型、知识库、数据库、文档解析结果。
- Word/PDF报告渲染、预览、下载和版本管理。

### 5.7 OCR识别模块

- 身份证、车牌、发票、合同识别。
- 自定义字段抽取配置。
- OCR结果结构化、置信度、原文位置和人工修订。

### 5.8 任务编排模块

- 异步任务创建、投递和消费。
- 任务状态机和阶段日志。
- 失败重试、手动重试、任务取消。
- 任务耗时统计和异常原因记录。

### 5.9 审计安全模块

- 操作审计日志。
- 敏感字段脱敏。
- 接口限流、防重复提交。
- 数据备份、访问追踪和异常告警。


### 5.10 前端应用模块

前端采用Vue 3 + TypeScript实现，按页面、组件、接口服务、状态管理和路由权限进行分层。

| 子模块 | 职责 |
| --- | --- |
| 路由与布局 | 登录页、主框架、菜单、面包屑、项目切换和权限路由 |
| 请求服务 | 统一封装HTTP请求、Token注入、requestId、错误提示和文件下载 |
| 状态管理 | 保存当前用户、当前项目、权限菜单、任务状态和页面缓存 |
| 通用组件 | 文件上传、分页表格、筛选表单、弹窗表单、状态标签、进度条、JSON查看器 |
| 业务页面 | 知识问答、合规审查、报告管理、OCR识别、知识库管理、系统管理 |
| 前端自测 | 每个页面负责人完成页面可用性、接口调用、异常提示和权限显示自测 |

前端不直接访问数据库、对象存储、大模型、OCR引擎或知识库服务，所有能力统一通过后端API调用。

## 6. 核心流程设计

### 6.1 知识问答流程

```text
用户提问
  ↓
参数校验与权限校验
  ↓
问题意图识别与路由判断
  ↓
知识库检索 / 数据库查询 / 大模型生成
  ↓
答案融合与来源引用
  ↓
返回答案并记录会话日志
```

### 6.2 合规审查流程

```text
上传文档与选择模板
  ↓
文件解析与文本抽取
  ↓
审查模板/规范匹配
  ↓
调用审查模型或规则服务
  ↓
生成问题清单和修改建议
  ↓
返回JSON结果并保存审查记录
```

### 6.3 报告生成流程

```text
创建报告生成配置
  ↓
创建异步生成任务
  ↓
文件准备与数据解析
  ↓
知识库检索与数据库查询
  ↓
模型生成报告内容
  ↓
内容审查
  ↓
模板渲染与PDF导出
  ↓
保存文件并更新任务状态
```

### 6.4 OCR识别流程

```text
上传图片或文档
  ↓
图像预处理
  ↓
OCR文字识别
  ↓
字段抽取与置信度计算
  ↓
结果结构化与权限过滤
  ↓
返回JSON并保存识别记录
```

## 8. 外部能力集成

### 8.1 大模型服务

- 通过HTTP接口调用。
- 支持模型名称、温度、最大Token数、超时时间配置。
- 支持请求、响应、耗时、错误信息记录。
- 支持失败重试和降级提示。

### 8.2 RAG知识库

- 支持项目知识库、政策标准库、行业资料库。
- 支持TopK、相似度阈值、领域过滤、标准编号过滤。
- 返回标题、来源、片段、分数和文档定位。

### 8.3 业务数据库

- 支持MySQL、PostgreSQL、人大金仓等关系型数据库。
- 默认只读访问。
- SQL必须经过安全校验和权限过滤。
- 慢查询需记录并设置超时。

### 8.4 OCR引擎

- 支持图片预处理、文字识别、字段抽取和置信度返回。
- 识别结果需要保留原文位置，便于人工复核。

## 9. 安全设计

### 9.1 认证授权

- 登录后签发JWT或会话令牌。
- 接口层统一校验认证状态。
- 业务层按项目、角色、资源类型进行授权。

### 9.2 数据隔离

- 所有业务数据包含projectId。
- 查询必须自动追加项目权限过滤。
- 文件访问、知识库检索、报告查看均需校验项目权限。

### 9.3 敏感信息保护

- 身份证号、手机号、银行卡号等敏感字段脱敏展示。
- 文件下载地址设置有效期。
- 外部服务调用不得记录明文密钥。

### 9.4 审计追踪

- 记录操作人、操作类型、对象类型、对象ID、IP、requestId、操作时间。
- 关键操作包括上传、下载、生成、审查、删除、分享、权限变更。

## 11. 异常处理设计

| 异常类型 | 处理方式 |
| --- | --- |
| 参数错误 | 返回参数错误码和字段说明 |
| 无权限 | 返回403错误，不暴露资源详情 |
| 文件解析失败 | 记录失败原因，允许重新解析 |
| 模型调用失败 | 记录请求摘要和错误原因，支持重试 |
| 知识库检索失败 | 降级为模型回答或返回明确提示 |

| 报告渲染失败 | 标记任务失败，保留阶段日志 |

## 12. 可观测性设计

- 所有请求生成requestId。
- 接口日志记录请求路径、用户、项目、耗时和状态码。
- 任务日志记录每个阶段的输入摘要、输出摘要、耗时和错误信息。
- 外部能力调用记录服务名称、请求摘要、响应摘要、耗时和状态。
- 监控慢接口、失败率、任务积压量、模型调用耗时和存储容量。


## 13. 前端架构设计

### 13.1 技术选型

| 技术 | 用途 |
| --- | --- |
| Vue 3 | 前端应用框架 |
| TypeScript | 类型约束和可维护性提升 |
| Vue Router | 路由管理和权限路由 |
| Pinia | 全局状态管理 |
| Axios | HTTP请求封装 |
| Element Plus或同类组件库 | 后台管理类基础组件 |
| Vite | 前端构建工具 |

### 13.2 前端目录建议

```text
src/
  api/              接口请求封装
  assets/           静态资源
  components/       通用组件
  layouts/          页面布局
  router/           路由配置和权限守卫
  stores/           状态管理
  utils/            工具函数
  views/            业务页面
    dashboard/      首页工作台
    project/        项目管理
    knowledge/      知识库管理
    qa/             知识问答
    review/         合规审查
    report/         报告管理
    ocr/            OCR识别
    system/         系统管理
```

### 13.3 前后端交互约定

- 前端所有请求统一从 `src/api` 发起，不在页面中散落拼接URL。
- 请求头统一注入 `Authorization` 和 `X-Request-Id`。
- 401统一跳转登录页，403统一展示无权限提示。
- 文件下载统一处理文件名、Blob和错误响应。
- 分页参数统一使用 `pageNo`、`pageSize`。
- 耗时任务页面通过任务详情接口轮询或SSE/WebSocket获取进度。

### 13.4 前端安全约束

- Token不写入URL。
- 无权限菜单和按钮不展示。
- 敏感字段按后端返回结果展示，不在前端自行还原。
- 上传文件前进行大小、类型和必填校验。
- 页面错误提示不展示后端堆栈信息。

### 14.2 服务边界

```text
Vue 3 + TypeScript 前端
        ↓ REST API
Java + Spring Boot 后端主系统
        ↓ REST API / 内部服务调用
Python智能算法服务
        ↓
大模型 / OCR引擎 / 向量库 / 文档解析组件
```

### 14.3 选型说明

- Java后端主系统负责稳定的企业级业务能力，包括权限、审计、任务、文件、报告、项目和接口治理。
- Python智能算法服务负责快速集成AI生态，包括大模型SDK、OCR框架、向量化模型、RAG流程和文档解析算法。
- Vue 3 + TypeScript负责前端工程和业务页面，提升组件复用、类型约束和页面维护效率。
- 前端不直接调用Python服务或数据库，所有业务访问统一经过Java后端。
- Java后端不直接耦合具体模型实现，通过适配层调用Python算法服务或第三方模型服务。


## 15. AI Adapter Implementation Update

The repository now includes `python-ai-service/` as the Python intelligent algorithm service skeleton. Qwen API keys are configured only in the Python service. The Java backend adds the `ai` module and calls Python through `app.ai.python-service.*` for model/Agent calls, RAG search, routing, context preparation, and database Q&A.

Database Q&A uses the safe pattern of Python-generated SQL/summaries and Java validation/execution. Java only executes read-only `SELECT`/`WITH` queries against enabled MySQL data sources in the current project, and blocks writes, DDL, multiple statements, and dangerous keywords.


## Advanced AI Adapter Update

The AI adapter now includes document indexing, chunking, embedding, vector retrieval, reranking, and a Python tool-calling Agent framework. The Python service supports `LOCAL`, `PGVECTOR`, and `MILVUS` vector providers. Qwen embeddings are used in production through `EMBEDDING_PROVIDER=QWEN`; Qwen rerank is configured through `RERANK_PROVIDER=QWEN`, `QWEN_RERANK_BASE_URL`, and `QWEN_RERANK_MODEL`; `LOCAL_HASH` is reserved for offline tests. Java exposes `/api/ai/knowledge/index` and `/api/ai/knowledge/search`, but Java does not directly access vector databases.

Database Q&A now supports MySQL, PostgreSQL, and Kingbase JDBC drivers while preserving strict read-only SQL validation, query timeout, and maximum-row limits.


PostgreSQL and Kingbase JDBC drivers are enabled for database Q&A execution. PostgreSQL can be integration-tested with `AI_TEST_POSTGRES_JDBC_URL`; Kingbase can be integration-tested with `AI_TEST_KINGBASE_JDBC_URL` when an available Kingbase test instance exists.
