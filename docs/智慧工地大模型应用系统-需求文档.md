# 智慧工地大模型应用系统-需求文档

文档说明：本文档为智慧工地大模型应用系统产品需求文档（PRD），用于指导产品设计、系统开发、接口联调和功能验收。本文档仅聚焦系统开发范围，不包含市场调研、课题研究、专利软著申报等非系统建设内容。

版本：v1.0 | 状态：草稿 | 最后更新：2026-07-02

## 一、项目背景与目标

### 1.1 背景

传统工地管理存在数据分散、资料标准不统一、规范审查依赖人工、报告编制效率低、现场证照票据识别成本高等问题。智慧工地大模型应用系统通过整合互联网政策资讯、本地知识库、业务系统数据库和现场文档图片资料，提供知识问答、合规审查、报告生成和OCR识别等能力，提升工地管理的智能化、标准化和自动化水平。

系统建设重点不是单纯训练一个模型，而是围绕智慧工地业务场景建设一套可落地的应用系统，使模型、知识库、数据库、文档处理和业务权限形成闭环。

### 1.2 建设目标

- 建设统一的智慧工地大模型应用入口，支撑知识问答、合规审查、报告生成和OCR识别。
- 支持Word、PDF、Excel、PPT、图片等多格式资料接入，并形成项目级知识库。
- 支持互联网政策资讯、本地文档、业务数据库等多源数据接入和统一检索。
- 支持按工地项目进行知识库隔离、数据权限控制和操作审计。
- 支持基于模板的合规审查和报告生成，输出结构化JSON、Word和PDF结果。
- 支持身份证、车牌、发票、合同及自定义字段的OCR识别。
- 支持任务异步处理、过程追踪、失败重试和结果留痕。

### 1.3 成功指标

| 指标 | 目标值 | 说明 |
| --- | --- | --- |
| 问答准确率 | >= 90% | 基于知识库、数据库和模型综合回答的业务准确率 |
| 合规审查准确率 | >= 95% | 对上传文档问题识别和修改建议的准确率 |
| OCR识别准确率 | >= 95% | 身份证、车牌、发票、合同等典型样本识别准确率 |
| 平均响应时间 | <= 3秒 | 普通问答、普通审查请求的平均响应时间 |
| P95响应时间 | <= 5秒 | 95百分位响应时间 |
| 标准报告生成时间 | <= 20秒 | 标准模板和中等数据量场景 |
| 复杂报告生成时间 | <= 30秒 | 多数据源、多章节、多图表报告场景 |
| 数据标准化完成率 | 100% | 入库数据必须经过统一清洗和标准化 |
| 数据缺失率 | <= 5% | 清洗后可用于模型和检索的数据缺失率 |
| 系统可用率 | >= 99.5% | 按月统计系统可用性 |

## 二、用户角色

| 角色 | 典型用户 | 核心诉求 | 权限范围 |
| --- | --- | --- | --- |
| 平台管理员 | 系统管理员、运维人员 | 管理用户、角色、项目、数据源和系统配置 | 全部管理权限 |
| 项目管理员 | 工地项目负责人 | 管理项目知识库、模板、报告和成员权限 | 本项目管理权限 |
| 业务人员 | 安全员、质量员、资料员、项目经理 | 使用问答、审查、报告和OCR能力处理日常资料 | 本项目业务操作权限 |
| 审核人员 | 合规审核、质量审核人员 | 查看审查结果、修改建议和报告内容 | 审核与确认权限 |
| 普通查看者 | 管理层、甲方查看人员 | 查看问答结果、报告和统计信息 | 只读查看权限 |

### 3.9 前端体验要求

- 前端采用Vue 3 + TypeScript开发。
- 页面需适配桌面端浏览器，关键操作在常见分辨率下不出现遮挡和横向溢出。
- 文件上传、报告生成、OCR识别、知识库入库等耗时操作必须展示进度或状态。
- 接口异常需给出明确提示，不直接暴露后端堆栈或技术错误。
- 表单提交需做前端必填、格式和长度校验。
- 无权限菜单和按钮不展示；直接访问无权限路由时进入无权限提示页。
- 对报告、审查、OCR等结果页提供复制、下载或导出入口。

## 四、非功能需求

### 4.1 性能要求

- 普通接口平均响应时间不超过3秒。
- P95响应时间不超过5秒。
- OCR单文档识别不超过5秒。
- 标准报告生成不超过20秒，复杂报告生成不超过30秒。
- 数据处理时间不超过2小时/数据集。

### 4.2 安全要求

- 所有非公开接口必须经过身份认证。
- 用户只能访问授权项目内的数据。
- 文件预览和下载必须校验权限。
- 身份证号、手机号、合同金额等敏感字段按权限脱敏展示。
- 关键操作必须记录审计日志。

### 4.3 可用性要求

- 系统可用率不低于99.5%。
- 外部模型、知识库、数据库不可用时，系统应返回明确错误信息并记录失败原因。
- 耗时任务不得长时间阻塞HTTP请求，应采用异步任务处理。

### 4.4 可维护性要求

- 系统按模块划分，模块边界清晰。
- 前端工程按页面、组件、接口服务、状态管理和路由权限分层组织。
- 接口返回结构统一，错误码统一管理。
- 日志包含requestId、用户、项目、任务、耗时和错误信息。
- 数据库结构变更需有迁移脚本和版本记录。

## 五、业务流程

### 5.1 知识问答流程

1. 用户选择项目并输入问题。
2. 系统识别问题类型和数据来源。
3. 根据路由策略调用模型、知识库、数据库或混合能力。
4. 系统生成答案，并附带引用来源。
5. 用户可继续追问或对回答进行反馈。

### 5.2 合规审查流程

1. 用户选择审查模板或标准。
2. 用户上传待审查文档或选择业务数据。
3. 系统解析文档并调用审查能力。
4. 系统返回问题清单、定位信息和修改建议。
5. 用户下载或保存审查结果。

### 5.3 报告生成流程

1. 用户选择报告类型和报告模板。
2. 用户选择知识库、数据源和参考文档。
3. 系统创建报告生成任务。
4. 后台任务执行数据准备、知识检索、模型生成、内容审查和模板渲染。
5. 系统生成Word/PDF报告并提供预览和下载。

### 5.4 OCR识别流程

1. 用户选择识别类型并上传图片或文档。
2. 系统执行OCR识别和字段抽取。
3. 系统返回结构化JSON结果。
4. 用户可编辑结果并下载或提交到业务系统。

## 七、技术选型要求

### 7.1 开发语言

| 系统部分 | 开发语言/技术 | 说明 |
| --- | --- | --- |
| 后端主系统 | Java + Spring Boot | 负责用户权限、项目管理、文件管理、知识库管理、知识问答、合规审查、报告生成、OCR结果管理、任务编排、审计日志和统一接口服务 |
| 前端应用 | Vue 3 + TypeScript | 负责登录、首页工作台、项目切换、知识库管理、知识问答、合规审查、报告管理、OCR识别、系统管理等页面 |
| 智能算法服务 | Python | 负责大模型调用、RAG检索、Embedding向量化、OCR识别、文档解析、智能路由等AI能力封装 |
| 数据库 | MySQL | 负责业务元数据、项目数据、权限数据、任务数据和审计日志存储 |
| 缓存/任务队列 | Redis | 负责缓存、异步任务队列、分布式锁和任务状态加速 |
| 文件存储 | MinIO | 负责文档、图片、模板、报告等对象文件存储 |
| 向量检索 | Milvus或pgvector | 负责知识库向量检索和RAG召回 |

### 7.2 集成方式

- 前端通过REST API调用Java后端主系统。
- Java后端负责业务编排、权限控制、数据管理和统一接口输出。
- Python智能算法服务通过REST API或内部服务接口与Java后端集成。
- 大模型、OCR、向量库、业务数据库等能力不直接暴露给前端。
- 所有跨服务调用必须记录请求摘要、响应摘要、耗时、状态和错误信息。

### 7.3 技术选型原则

- 企业业务系统能力优先放在Java后端，保证权限、安全、事务和可维护性。
- AI算法、OCR、RAG和向量化能力优先放在Python服务，方便模型生态集成和算法迭代。
- 前端采用Vue 3 + TypeScript，保证页面开发效率、类型约束和组件复用。
- 存储组件优先选择成熟、可本地化部署、便于运维的开源方案。
